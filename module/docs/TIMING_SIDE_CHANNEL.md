# タイミングサイドチャネル検知ベクター

## 1. 検知原理

タイミングサイドチャネル検知は、ハードウェアTEE（Trusted Execution Environment）による暗号操作と、ソフトウェアシミュレーションによる暗号操作の実行時間差を統計的に分析する手法である。

検知の基本原理は以下の通り：

1. **基準測定**: 正規のハードウェアTEEによるattestation操作の実行時間分布を収集する
2. **対象測定**: 検査対象デバイスで同一操作の実行時間分布を収集する
3. **統計比較**: 両分布の差異を統計的に評価し、閾値を超える場合にシミュレーションと判定する

ハードウェアTEEは専用のセキュアプロセッサ上で動作するため、固有の実行時間特性を持つ。ソフトウェアシミュレーションはメインプロセッサ上で動作するため、以下の点で異なる時間特性を示す：

- キャッシュヒット/ミスパターンの違い
- コンテキストスイッチによる遅延変動
- メモリアクセスパターンの違い
- CPUスケジューリングの影響

## 2. 検知例

以下は実際のタイミングサイドチャネル検知ログの例である：

```
Register timer bound_cpu0 attested 0.932ms non-attested 1.068ms diff -0.136ms ratio 1.146x filteredBadSamples=20/500 threshold > 1.1x Positive
```

このログの各フィールドの意味：

| フィールド | 値 | 説明 |
|---|---|---|
| bound_cpu0 | - | CPU0にピン留めした状態での測定 |
| attested | 0.932ms | ハードウェアTEEによる実行時間 |
| non-attested | 1.068ms | シミュレーションによる実行時間 |
| diff | -0.136ms | 時間差（attested - non-attested） |
| ratio | 1.146x | 比率（non-attested / attested） |
| filteredBadSamples | 20/500 | 外れ値として除外されたサンプル数 |
| threshold | > 1.1x | 検知閾値 |
| 判定 | Positive | シミュレーション検知 |

## 3. 根本的限界

ソフトウェアシミュレーションがハードウェアTEEと完全に不可区別になれない根本的理由：

### 3.1 実行環境の本質的差異

ハードウェアTEE（TrustZone、Titan M等）は物理的に分離されたプロセッサまたはセキュアワールドで動作する。この物理的分離により：

- **固定レイテンシ**: セキュアワールドへの遷移（SMC呼び出し）は一定のオーバーヘッドを持つ
- **独立したキャッシュ**: セキュアワールドは独自のキャッシュ階層を持ち、ノーマルワールドの影響を受けない
- **割り込み非干渉**: セキュア処理中はノーマルワールドの割り込みが遮断される

### 3.2 統計的検出可能性

十分なサンプル数（N > 100）を収集すれば、以下の統計的特徴の差異が検出可能：

- 実行時間の分散（ハードウェアTEEの方が一般的に小さい）
- 分布の形状（ハードウェアTEEは正規分布に近い、シミュレーションは右裾が長い）
- 自己相関パターン（シミュレーションはCPU負荷に依存した相関を示す）

### 3.3 情報理論的限界

シミュレーションは本質的に追加の計算ステップを必要とするため、最小実行時間の下限がハードウェアTEEと異なる。この差異は原理的に除去不可能である。

## 4. 緩和策

### 4.1 Jitter Injection（ジッター注入）

**説明**: 各操作にランダムな遅延を追加し、実行時間の分布を意図的にぼかす。

**実装**:
```
waitTime = random(0, maxJitter)
nanosleep(waitTime)
```

**有効性**: 単純な平均値比較による検知を困難にする。

**限界**: 分散が増大するため、分散ベースの検知には逆効果となる場合がある。また、最小実行時間の下限は変わらないため、最小値ベースの検知には無効。

### 4.2 CPU Pinning（CPUピン留め）

**説明**: シミュレーション処理を特定のCPUコアに固定し、コア間マイグレーションによる遅延変動を排除する。

**実装**:
```
sched_setaffinity(0, sizeof(cpu_set_t), &cpuset)
```

**有効性**: コアマイグレーションに起因する外れ値を大幅に削減する。測定の再現性が向上する。

**限界**: 同一コア上の他プロセスとの競合は排除できない。また、ピン留め自体が検知シグナルとなる可能性がある。

### 4.3 Pre-Warming（プリウォーミング）

**説明**: 実際の操作前にダミー操作を実行し、キャッシュとブランチプレディクタを温める。

**実装**:
```
for (int i = 0; i < WARMUP_ITERATIONS; i++) {
    dummyOperation();
}
// 本番操作
realOperation();
```

**有効性**: コールドスタートによる初回遅延を排除し、安定した実行時間を実現する。

**限界**: ウォームアップ自体に時間がかかるため、全体のレイテンシが増加する。また、キャッシュエビクションが発生する環境では効果が限定的。

### 4.4 Latency Equalizer（レイテンシイコライザー）

**説明**: ハードウェアTEEの実行時間プロファイルを事前に収集し、シミュレーション結果の実行時間をそのプロファイルに合わせて調整する。

**実装**:
```
targetTime = referenceProfile.mean / detectionThreshold
waitTime = max(0, targetTime - elapsedTime)
jitter = random(-stddev, +stddev)
nanosleep(waitTime + jitter)
```

**有効性**: 平均値と分散の両方をハードウェアTEEのプロファイルに近づけることができる。4つの緩和策の中で最も効果的。

**限界**:
- 参照プロファイルがデバイス固有であるため、汎用的なプロファイルでは精度が低下する
- 実際の操作時間が目標時間を超えた場合、短縮は不可能
- 高次の統計量（尖度、歪度）の一致は困難
- プロファイル収集自体が検知される可能性がある

## 5. 結論

タイミングサイドチャネルに対する完全な不可区別性（indistinguishability）は、ソフトウェアシミュレーションでは原理的に達成不可能である。

その理由は：

1. **物理的差異**: ハードウェアTEEとメインプロセッサは物理的に異なる実行環境であり、この差異は完全には隠蔽できない
2. **情報理論的制約**: シミュレーションは追加の計算を必要とし、この計算コストはゼロにできない
3. **統計的検出力**: 十分なサンプル数があれば、いかなる緩和策も統計的に検出可能

ただし、上記の緩和策を組み合わせることで、検知に必要なサンプル数を大幅に増加させ、実用的な検知を困難にすることは可能である。これは「完全な不可区別性」ではなく「計算量的な検知困難性」を目指すアプローチである。
